# Review-Checkliste Modell

## Modelllogik

- [ ] Zweck des Modells ist klar beschrieben.
- [ ] Zielgruppe „Kleinstunternehmen bis 10 Mitarbeitende“ ist eindeutig.
- [ ] Branchenübergreifende Anwendung ist nachvollziehbar.
- [ ] Fokus auf Website, Google-Unternehmensprofil, Social Media, Newsletter, Content-Erstellung und einfache Automationen ist erkennbar.
- [ ] Die sechs Phasen sind logisch aufgebaut.
- [ ] Jede Phase hat Ziel, Input, KI-Aufgabe, menschliche Aufgabe und Ergebnis.
- [ ] Das Modell bleibt ein Beratungs-/Prozessframework und wird nicht zur App-, SaaS- oder Plattformidee.

## Praxisnutzen

- [ ] Das Modell ist als Praxis-Workbook nutzbar.
- [ ] Es gibt verständliche Worksheets.
- [ ] Es gibt ein kompaktes Canvas.
- [ ] Die Use-Case-Auswahl ist auch für Anfänger verständlich.
- [ ] Die Pilotlogik ist realistisch.
- [ ] Der Aufwand passt zu Kleinstunternehmen.
- [ ] Das Modell vermeidet Overengineering.

## Daten und Datenschutz

- [ ] Es werden keine echten Kundendaten verlangt.
- [ ] Es werden keine personenbezogenen E-Mails verwendet.
- [ ] Es werden keine Gesundheitsdaten verwendet.
- [ ] Es werden keine Bewerbungsdaten verwendet.
- [ ] Es werden keine vertraulichen Finanzdaten verwendet.
- [ ] Es werden keine internen Kundendokumente verwendet.
- [ ] Beispiele sind synthetisch oder ausreichend abstrahiert.
- [ ] Datenschutzgrenzen sind für Anfänger verständlich erklärt.

## Quellen und Annahmen

- [ ] Sakizli-Modelle als Framework-Referenzen sind benannt.
- [ ] Projekt-Scope / Repository-Standard ist berücksichtigt.
- [ ] Quellenstatus ist dokumentiert.
- [ ] Annahmen sind als Annahmen erkennbar.
- [ ] Es gibt keine erfundenen Quellen.
- [ ] Öffentliche Quellen werden nur als Link, Kurzreferenz oder eigene Zusammenfassung verwendet.

## Trust Layer

- [ ] `data_risk: yellow` ist plausibel.
- [ ] `human_review_required: true` ist gesetzt.
- [ ] `ai_act_proximity: transparency` oder `unclear` ist begründet.
- [ ] `required_review_level: trust_review` ist gesetzt.
- [ ] Disclaimer ist vorhanden.
- [ ] Keine Rechtsberatung wird behauptet.
- [ ] Kein Audit wird behauptet.
- [ ] Keine automatisierte Entscheidung über Menschen wird vorgesehen.

## Beispielanwendung

- [ ] Es gibt mindestens ein vollständiges Beispiel.
- [ ] Beispiel ist realitätsnah.
- [ ] Beispiel ist synthetisch.
- [ ] Beispiel zeigt Input, Ablauf, KI-Aufgabe und Ergebnis.
- [ ] Beispiel enthält menschliche Prüfung.
- [ ] Beispiel enthält Bewertung: positiv, nachbearbeitbar oder negativ.
- [ ] Beispiel zeigt mindestens einen Failure Mode oder eine Nachbesserung.

## Failure Modes

- [ ] Failure Modes sind konkret.
- [ ] Failure Modes beziehen sich auf kleine KMU.
- [ ] Datenschutzrisiken sind benannt.
- [ ] Halluzinationen / erfundene Quellen sind benannt.
- [ ] falsche Automatisierung ist benannt.
- [ ] fehlende menschliche Freigabe ist benannt.
- [ ] zu breiter Scope ist benannt.
- [ ] fehlende Erfolgsmessung ist benannt.
- [ ] Gegenmaßnahmen sind beschrieben.

## Statusentscheidung

### `draft`

Ankreuzen, wenn:

- [ ] wichtige Teile fehlen,
- [ ] Beispiel fehlt,
- [ ] Quellen fehlen,
- [ ] Failure Modes fehlen,
- [ ] Datenschutzgrenzen unklar sind.

### `bronze_candidate`

Ankreuzen, wenn:

- [ ] Grundstruktur vollständig ist,
- [ ] Zielgruppe klar ist,
- [ ] Worksheets vorhanden sind,
- [ ] Beispiel vorhanden ist,
- [ ] Failure Modes vorhanden sind,
- [ ] Trust Layer vorhanden ist,
- [ ] aber Peer Review oder Trust Review noch aussteht.

### `bronze`

Ankreuzen nur, wenn:

- [ ] alle Pflichtdateien vorhanden sind,
- [ ] Metadaten valide sind,
- [ ] Zweck und Zielgruppe klar sind,
- [ ] mindestens ein Beispiel vorhanden ist,
- [ ] Trust Layer vorhanden ist,
- [ ] keine sensiblen Echtdaten enthalten sind,
- [ ] Peer Review abgeschlossen ist,
- [ ] notwendige Trust-Prüfung abgeschlossen ist,
- [ ] Maintainer freigegeben hat.
